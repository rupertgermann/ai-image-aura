# OpenAI Request Mapping

This document maps AURA AI user actions to the request payloads generated by the browser client.

## Shared Defaults

All OpenAI image requests in this project use:

- `model: gpt-image-1.5`
- `n: 1`
- bearer token authorization from the locally stored API key

All Autopilot reasoning requests use:

- `model: gpt-5.4`
- `input` blocks composed from system instructions and user content
- bearer token authorization from the locally stored API key

The helper responsible for OpenAI request construction lives in `src/utils/openai.ts`.

## Generate View Request Mapping

### Prompt-only generation

When the Generate view has no reference images, the app sends a JSON request to the generations endpoint.

#### UI inputs

- Prompt textarea
- Quality selector
- Aspect ratio selector
- Background selector
- Style selector
- Lighting selector
- Palette selector

#### Derived prompt behavior

The app combines the base prompt with selected creative modifiers before sending the request.

Example pattern:

- base prompt
- optional style modifier
- optional lighting modifier
- optional `color palette: ...` modifier

#### Request shape

```json
{
  "model": "gpt-image-1.5",
  "prompt": "<base prompt with optional style, lighting, and palette modifiers>",
  "n": 1,
  "size": "1024x1024 | 1536x1024 | 1024x1536",
  "quality": "low | medium | high",
  "background": "opaque | transparent"
}
```

`size` and `background` are omitted when the UI is set to `auto`.

### Reference-based generation

When the Generate view includes uploaded reference images, the app switches to the edits endpoint and sends a `FormData` payload.

#### Request fields

- `model`
- `prompt`
- `n`
- one `image[]` field per uploaded reference file
- optional `size`
- optional `quality`
- optional `background`

This enables prompt-guided generation using visual references while keeping the same visible Generate controls.

## Editor View Request Mapping

The Editor view always starts from the current canvas content.

### AI transform flow

When the user runs an AI transform:

- the current canvas is exported to a PNG blob
- the blob is wrapped as `edit-input.png`
- the exported canvas file becomes the first `image[]` entry
- additional uploaded reference images are appended to the same request
- the app sends the request to `POST /v1/images/edits`

#### Request fields

- `model`
- `prompt`
- `n`
- `image[]` entries for the current canvas and any optional references
- `quality: medium`

Editor transforms do not expose size or background controls.

### Save flows after an edit

After a successful edit response:

- `Save Changes` overwrites the current archive image locally
- `Save as Copy` creates a new archive image locally
- lineage metadata is written locally based on the action and edit context

These save paths do not issue additional OpenAI requests.

## Autopilot Request Mapping

Autopilot combines image and Responses API calls into a loop.

### Goal translation

The `Translate to Prompt` action sends a Responses API request with:

- a system prompt that asks for one concise image-generation prompt
- user text in the form `Creative goal:\n...`

The returned text replaces the current Generate prompt.

### Iteration loop

Each Autopilot iteration performs:

1. An image request using the current Generate settings and reference images
2. A Responses API evaluation request that includes:
   - the creative goal as text
   - the generated image as `input_image`
3. A Responses API refinement request that includes:
   - the creative goal
   - the current prompt
   - evaluator feedback as bullet-style text

The refinement step is skipped after the final iteration or when the satisfaction threshold is met.

### Iteration metadata

For each completed iteration, the app records local lineage metadata containing:

- `goalText`
- `iterationNumber`
- `evaluatorScore`
- `evaluatorFeedback`
- the prompt and generation settings used for that iteration
- `outputImageDataUrl` for replay and timeline preview

## Response Handling

For image requests, the client expects:

```json
{
  "data": [
    {
      "b64_json": "..."
    }
  ]
}
```

For Responses API requests, the client accepts either:

- a top-level `output_text` field
- `output[].content[]` entries with `type: "output_text"`

Image payloads are converted into browser data URLs and then:

- previewed in the current view
- saved locally when the user archives the image
- reused in archive detail and editor flows as needed

Responses API text is trimmed and then used as:

- a translated prompt
- parsed evaluation JSON
- a refined prompt

## Current Limitations

- The app requests one image per generation or edit operation
- The app does not use streaming responses
- The app does not request masks or variations
- The app does not expose output format or compression controls
- Editor transforms always use `quality: medium`
