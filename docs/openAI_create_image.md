# OpenAI Request Mapping

This document maps AURA AI user actions to the request payloads generated by the browser client.

## Shared Defaults

All OpenAI requests in this project currently use:

- `model: gpt-image-1.5`
- `n: 1`
- bearer token authorization from the locally stored API key

The helper responsible for request construction lives in `src/utils/openai.ts`.

## Generate View Request Mapping

### Prompt-only generation

When the Generate view has no reference images, the app sends a JSON request to the generations endpoint.

#### UI inputs

- Prompt textarea
- Quality selector
- Aspect ratio selector
- Background selector
- Style selector
- Lighting selector
- Palette selector

#### Derived prompt behavior

The app combines the base prompt with selected creative modifiers before sending the request.

Examples:

- Base prompt only
- Base prompt plus selected style
- Base prompt plus style, lighting, and palette descriptors

#### Request shape

```json
{
  "model": "gpt-image-1.5",
  "prompt": "<base prompt with optional style, lighting, and palette modifiers>",
  "n": 1,
  "size": "1024x1024 | 1536x1024 | 1024x1536",
  "quality": "low | medium | high",
  "background": "opaque | transparent"
}
```

`size` and `background` are omitted when the UI is set to `auto`.

### Reference-based generation

When the Generate view includes uploaded reference images, the app switches to the edits endpoint and sends a `FormData` payload.

#### Request fields

- `model`
- `prompt`
- `n`
- `image[]` for each uploaded reference file
- optional `size`
- optional `quality`
- optional `background`

This enables prompt-guided generation using visual references.

## Editor View Request Mapping

The Editor view always starts from the current canvas content.

### AI transform flow

When the user runs an AI transform:

- the current canvas is exported to a PNG blob
- the blob is wrapped as `edit-input.png`
- the exported canvas file becomes the first `image[]` entry
- additional uploaded reference images are appended to the same request
- the app sends the request to `POST /v1/images/edits`

#### Request fields

- `model`
- `prompt`
- `n`
- `image[]` entries for the current canvas and any optional references
- `quality: medium`

Editor transforms do not currently expose size or background controls.

## Response Handling

For both flows, the client expects:

```json
{
  "data": [
    {
      "b64_json": "..."
    }
  ]
}
```

The returned base64 payload is converted into a browser data URL and then:

- previewed in the current view
- saved locally when the user archives the image
- reused in the editor or archive as needed

## Current Limitations

- The app requests one image per operation
- The app does not use streaming responses
- The app does not request masks or variations
- The app does not use the Responses API for image generation
- Output format and compression controls are not exposed in the UI
